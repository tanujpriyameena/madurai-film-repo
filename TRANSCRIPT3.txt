● Yes, this will work on Netlify with one important caveat.

  The good: Create React App with react-scripts build is exactly what Netlify
  expects. Your build command is npm run build, output directory is build/ —
  Netlify auto-detects both. localStorage is purely client-side, so no server is
   needed at all.

  The one catch — React Router: When you add React Router, direct URL access to
  routes like /collections or /movie/5 will return a 404 on Netlify, because
  Netlify looks for actual files at those paths and finds none. The fix is
  simple: add a _redirects file to your public/ folder with this single line:

  /*    /index.html   200

  That tells Netlify to serve index.html for all routes and let React Router
  take over. Without it, only / will work reliably.

  Everything else in your plan — static JSON data, localStorage, client-side
  filtering — is fully Netlify-compatible with no changes needed.

● Good catch. Here's the practical breakdown:

  Copyright situation by film:
  - Casablanca (1942) — the original poster is almost certainly public domain.
  US works published before 1964 that weren't copyright-renewed have entered the
   public domain, and most film posters from that era qualify. You're likely
  fine here.
  - Cars (2006) / Avatar (2025) — actively copyrighted by Disney/Pixar and 20th
  Century Studios respectively. Wikipedia's use is permitted under "fair use"
  for encyclopedic identification, but that exemption is specific to Wikipedia
  and doesn't extend to your site.

  Practical solutions:

  1. TMDB (The Movie Database) — the cleanest option. It's a free API that
  provides official poster images with clear terms allowing personal and
  educational projects. You'd fetch posters via their API rather than hosting
  images locally. This is how most movie apps handle it.
  2. Keep placeholders for copyrighted posters — for a class assignment, a
  styled placeholder is perfectly acceptable and sidesteps the issue entirely.
  3. Draw/design your own placeholder art — zero copyright risk, and counts as
  original work.

  On credits: Attribution is good practice and shows academic integrity, but it
  does not resolve copyright on its own. Copyright and credit are separate
  issues — you can credit a source and still be infringing.

  For a midterm project, my recommendation would be to either use TMDB for
  posters or keep placeholders, and note in your README that poster sourcing was
   intentionally deferred due to copyright. Want me to set up TMDB integration,
  or keep it simple with styled placeholders for now?

❯ How to use TMDB posters, please explain. 

● Here's how it works:

  1. Get an API key
  - Create a free account at https://www.themoviedb.org
  - Go to Settings → API → Request an API key (choose "Developer")
  - You'll receive a key that looks like a1b2c3d4e5f6...

  2. Find a movie's poster path
  TMDB has a search endpoint. For example, searching "Casablanca" returns a JSON
   response that includes a field like:
  "poster_path": "/5K7cOHoay2mZusSLezcs5R7x0GN.jpg"
  That path alone is not a usable URL — it's just a filename.

  3. Construct the full image URL
  TMDB serves images through their CDN. You prepend a base URL and a size:
  https://image.tmdb.org/t/p/w500/5K7cOHoay2mZusSLezcs5R7x0GN.jpg
  Common sizes are w200, w300, w500, original. For movie cards, w500 is
  standard.

  4. How this fits into your project
  You have two options:
  - Static — look up the poster path for each of your three movies manually on
  TMDB, build the full URL, and paste it directly into movies.js. No API calls
  at runtime, no key exposed.
  - Dynamic — fetch poster data from the TMDB API at runtime inside React.
  Requires storing your API key, which needs care (environment variables via
  .env).

  For your project, the static approach is the right call. You only have three
  movies, so you'd just look each one up once on TMDB, grab the poster_path,
  build the URL, and drop it into movies.js. No API key in your code, no runtime
   fetching, no complexity.

  Want me to walk you through finding the paths, or would you prefer to do that
  yourself and just hand me the URLs to put in?

❯ Can you explain to me how localStorage works in this case? If I add a     
  rating and a review, how does it persist? If say, I reload the page, what 
   happens? I like the functionality by the way, I'm just curious how this  
  is working right now.                                                     

● Good question. Here's how it works end to end.

  What localStorage is
  It's a simple key-value store built into the browser itself — not your server,
   not React, just the browser. Data written there survives page reloads, tab
  closes, and even full browser restarts. It only clears if the user manually
  clears their browser data, or if you call localStorage.removeItem() in code.

  What your app stores
  Two keys in localStorage:
  "ratings"  →  { "1": 4, "2": 5 }
  "reviews"  →  { "1": "Great film.", "3": "A classic." }
  Each is a JSON string. The keys inside are movie IDs, so each film has its own
   slot.

  What happens when you rate a film
  1. You click a star
  2. handleRate runs — it reads the current ratings object out of localStorage,
  sets stored[id] = star, then writes the whole thing back
  3. React state updates so the stars reflect the new rating immediately

  What happens on reload
  The useState for both rating and review uses an initializer function:
  const [rating, setRating] = useState(() => {
    const stored = JSON.parse(localStorage.getItem('ratings') || '{}');
    return stored[id] || 0;
  });
  That function runs once when the component first mounts. It reads from
  localStorage before React renders anything, so the stars and review text are
  already correct the moment the page loads — there's no flicker or loading
  state.

  The limitation
  Since it's stored in the browser, it's device-specific. If you open the site
  on a different computer or a different browser, the ratings and reviews won't
  be there. That's fine for a personal tracker — it's by design. A backend
  database would be needed to sync across devices, but that's well beyond the
  scope of this project.

● How the click works, step by step:

  When you click ♡ on a MovieCard:
  1. handleFavorite fires, calling e.stopPropagation() first — this tells the
  browser "don't bubble this click up to the <Link> wrapper," so you don't
  accidentally navigate to the movie page
  2. It calls toggleFavorite(movie.id) in App.js
  3. toggleFavorite checks the current favorites array — if the ID is already in
   it, it filters it out; if not, it appends it
  4. The updated array is written to localStorage, and React state is updated in
   the same step
  5. Because favorites lives in App.js, every component that received it as a
  prop re-renders instantly with the new value — the heart fills in on the card,
   the movie appears on the Profile page, all without a page reload

  ---
  Why not just a heart icon with local state?

  A local heart icon — say, useState(false) inside MovieCard — would look like
  it works. Click it, heart fills. But:

  - That state lives and dies with the card component. Navigate away and come
  back, it resets
  - The Profile page has no way to know which movies were favorited — it can't
  share local state across components
  - You'd have to do extra work reading from localStorage on every mount just to
   compensate

  By lifting state to App.js, there's one source of truth. Every component reads
   from the same array, so they're always in sync — toggle in Collections, it's
  already reflected on Profile. That's the core React principle: shared state 
  belongs at the common ancestor.

● Great question to ask before touching any code. Here's the full picture.

  ---
  What Firebase is

  Firebase is a platform by Google that provides backend services — auth,
  database, storage — without you writing a server. You connect to it directly
  from your React app using their SDK. For your project, you need two services:

  Firebase Authentication — handles sign up, login, logout, and session
  persistence entirely. It issues each user a unique uid. You never store
  passwords yourself.

  Firestore — a cloud NoSQL database. Data is stored as documents inside
  collections. Think of it like a deeply nested JSON object that lives in the
  cloud, with real-time sync and security rules built in.

  ---
  How it fits into your project

  Right now everything is keyed to the device via localStorage. With Firebase,
  the key becomes the user's uid instead. A typical Firestore structure for your
   app would look like:

  users/
    {uid}/
      ratings/
        {movieId}: 4
      reviews/
        {movieId}: "Great film."
      favorites: [1, 3]
      watched: [2]
      bio: "..."

  Each user's data is completely isolated under their own uid document.

  ---
  How Auth works in practice

  1. User signs up → Firebase creates an account and returns a user object with
  a uid
  2. User logs in → same user object is returned
  3. onAuthStateChanged is a listener you set up once in your app — it fires on
  every page load and tells you immediately whether a user is logged in or not.
  This is how session persistence works — Firebase handles the token storage
  automatically
  4. Logout → Firebase clears the session

  ---
  How Firestore reads and writes work

  Instead of localStorage.setItem('ratings', ...) you do something like:
  // Write
  await setDoc(doc(db, 'users', uid, 'ratings', movieId), { value: 4 });

  // Read
  const snap = await getDoc(doc(db, 'users', uid, 'ratings', movieId));

  Firestore also supports real-time listeners, but for your app simple
  reads/writes are enough.

  ---
  The security rules piece (Phase 8 step 21)

  Firestore lets you write rules that run server-side before any read or write
  is allowed. Your rules would say:
  - A user can only read/write their own data (request.auth.uid == userId)
  - Exception: reviews and ratings are readable by anyone (for a public listing
  later)

  This means even if someone tried to call the API directly, they couldn't touch
   another user's favorites or bio.

  ---
  What the migration from localStorage looks like

  The good news is your data model is already clean. The localStorage keys you
  have (ratings, reviews, favorites, watched, bio) map directly to Firestore
  subcollections or fields. Phase 8 is mostly a find-and-replace of localStorage
   calls with Firestore calls, wrapped in async/await.

  ---

  1. Create a Firebase project at https://console.firebase.google.com
  2. Enable Authentication (Email/Password provider)
  3. Enable Firestore (start in test mode, then tighten security rules)
  4. Register your web app inside the project — Firebase gives you a config
  object with your API keys
  5. Install the Firebase SDK: npm install firebase

  Once you have the config object, we can start Phase 7. Want to proceed with
  setup?

● Sure. Here's how it all fits together.

  ---
  Firebase Auth handles the actual account management

  When you call signup(email, password), Firebase creates the account on their
  servers and returns a user object. When you call login(email, password),
  Firebase verifies the credentials and returns the same object. You never touch
   passwords directly — Firebase handles hashing, storage, and verification
  entirely.

  The user object Firebase gives you contains fields like uid (a unique ID for
  that user), email, and displayName. That's where the name lives — not in your
  own database, just in Firebase's auth system.

  ---
  onAuthStateChanged is the backbone of session persistence

  This is a listener you set up once in AuthContext.js. Firebase calls it
  automatically whenever the auth state changes — on login, logout, and
  crucially, on every page load. Firebase stores an auth token in the browser,
  so when the page refreshes, onAuthStateChanged fires immediately with the
  logged-in user. You don't have to do anything extra for persistence — it's
  built in.

  The loading state in AuthContext exists specifically for this: the app waits
  until that first onAuthStateChanged fires before rendering anything, so
  there's never a flash where the app thinks nobody is logged in right after a
  refresh.

  ---
  Why React Context?

  currentUser needs to be accessible in Navbar, Movie, MovieCard, Profile, and
  ProtectedRoute — components at very different levels of the tree. Without
  Context, you'd have to pass it as props through every layer. Context makes it
  available anywhere with a single useAuth() call.

  ---
  ProtectedRoute is just a conditional render

  return currentUser ? children : <Navigate to="/login" />;

  That's the entire thing. If Firebase says there's a logged-in user, render the
   page. If not, redirect to login. Because onAuthStateChanged runs before the
  app renders (thanks to the loading guard), this check is always accurate on
  page load — it never incorrectly bounces a logged-in user to the login page.

  ---
  What's not happening yet (Phase 8)

  Right now, favorites, watched, ratings, reviews, and bio are still in
  localStorage — meaning they're device-specific and not tied to the Firebase
  account. Phase 8 moves all of that into Firestore, keyed by uid, so each user
  sees only their own data from any device.

● Yes, it will work on Netlify without any extra configuration, for two reasons:

  Firebase is called from the browser, not the server. Your firebase.js config
  and all the Firestore/Auth calls run entirely in the user's browser — there's
  no server component that Netlify needs to set up. Netlify just serves the
  built React files; the browser handles all Firebase communication directly.

  The config is already in the code. When Netlify runs npm run build, it bundles
   firebase.js (which is gitignored but will need to be handled — more on that
  below) into the final output. The API keys are what connect the app to your
  Firebase project, and they travel with the bundle.

  ---
  One thing to sort out before you push for Netlify deployment:

  Since src/firebase.js is in .gitignore, it won't be on GitHub, so Netlify's
  build will fail because the import won't resolve.

  The standard solution is Netlify environment variables. Instead of committing
  the file, you:
  1. Store each Firebase config value as an environment variable in Netlify's
  dashboard (Site Settings → Environment Variables)
  2. Replace firebase.js with a version that reads from process.env.REACT_APP_*
  variables

  That's a small change we can make when you're ready to deploy. No need to do
  it now — just something to be aware of before the push.

❯ So, just to confirm again. Since this is all local, can I simply close    
  out of localhost, without consequence? Or should I be touching firebase   
  and shutting something down? Just concerned that something would break if 
   I casually close localhost and don't touch it for a day.                 

● You can close localhost without any concern. Just stop the npm start process
  (Ctrl+C in the terminal) and that's it.

  Nothing needs to be "shut down" on the Firebase side. Firebase Auth and
  Firestore are always-on cloud services — they don't have a session tied to
  your local dev server. Your data in Firestore persists indefinitely, your user
   accounts stay intact, and everything will be exactly as you left it when you
  run npm start again tomorrow.

  The only thing that lives locally is the dev server itself, and that's
  stateless — it just serves files. Stopping it has no effect on Firebase or
  your data.
