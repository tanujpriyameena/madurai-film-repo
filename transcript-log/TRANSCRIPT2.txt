● Yes, this will work on Netlify with one important caveat.

  The good: Create React App with react-scripts build is exactly what Netlify
  expects. Your build command is npm run build, output directory is build/ —
  Netlify auto-detects both. localStorage is purely client-side, so no server is
   needed at all.

  The one catch — React Router: When you add React Router, direct URL access to
  routes like /collections or /movie/5 will return a 404 on Netlify, because
  Netlify looks for actual files at those paths and finds none. The fix is
  simple: add a _redirects file to your public/ folder with this single line:

  /*    /index.html   200

  That tells Netlify to serve index.html for all routes and let React Router
  take over. Without it, only / will work reliably.

  Everything else in your plan — static JSON data, localStorage, client-side
  filtering — is fully Netlify-compatible with no changes needed.

● Good catch. Here's the practical breakdown:

  Copyright situation by film:
  - Casablanca (1942) — the original poster is almost certainly public domain.
  US works published before 1964 that weren't copyright-renewed have entered the
   public domain, and most film posters from that era qualify. You're likely
  fine here.
  - Cars (2006) / Avatar (2025) — actively copyrighted by Disney/Pixar and 20th
  Century Studios respectively. Wikipedia's use is permitted under "fair use"
  for encyclopedic identification, but that exemption is specific to Wikipedia
  and doesn't extend to your site.

  Practical solutions:

  1. TMDB (The Movie Database) — the cleanest option. It's a free API that
  provides official poster images with clear terms allowing personal and
  educational projects. You'd fetch posters via their API rather than hosting
  images locally. This is how most movie apps handle it.
  2. Keep placeholders for copyrighted posters — for a class assignment, a
  styled placeholder is perfectly acceptable and sidesteps the issue entirely.
  3. Draw/design your own placeholder art — zero copyright risk, and counts as
  original work.

  On credits: Attribution is good practice and shows academic integrity, but it
  does not resolve copyright on its own. Copyright and credit are separate
  issues — you can credit a source and still be infringing.

  For a midterm project, my recommendation would be to either use TMDB for
  posters or keep placeholders, and note in your README that poster sourcing was
   intentionally deferred due to copyright. Want me to set up TMDB integration,
  or keep it simple with styled placeholders for now?

❯ How to use TMDB posters, please explain. 

● Here's how it works:

  1. Get an API key
  - Create a free account at https://www.themoviedb.org
  - Go to Settings → API → Request an API key (choose "Developer")
  - You'll receive a key that looks like a1b2c3d4e5f6...

  2. Find a movie's poster path
  TMDB has a search endpoint. For example, searching "Casablanca" returns a JSON
   response that includes a field like:
  "poster_path": "/5K7cOHoay2mZusSLezcs5R7x0GN.jpg"
  That path alone is not a usable URL — it's just a filename.

  3. Construct the full image URL
  TMDB serves images through their CDN. You prepend a base URL and a size:
  https://image.tmdb.org/t/p/w500/5K7cOHoay2mZusSLezcs5R7x0GN.jpg
  Common sizes are w200, w300, w500, original. For movie cards, w500 is
  standard.

  4. How this fits into your project
  You have two options:
  - Static — look up the poster path for each of your three movies manually on
  TMDB, build the full URL, and paste it directly into movies.js. No API calls
  at runtime, no key exposed.
  - Dynamic — fetch poster data from the TMDB API at runtime inside React.
  Requires storing your API key, which needs care (environment variables via
  .env).

  For your project, the static approach is the right call. You only have three
  movies, so you'd just look each one up once on TMDB, grab the poster_path,
  build the URL, and drop it into movies.js. No API key in your code, no runtime
   fetching, no complexity.

  Want me to walk you through finding the paths, or would you prefer to do that
  yourself and just hand me the URLs to put in?

   happens? I like the functionality by the way, I'm just curious how this

● Good question. Here's how it works end to end.

  What localStorage is
  It's a simple key-value store built into the browser itself — not your server,
   not React, just the browser. Data written there survives page reloads, tab
  closes, and even full browser restarts. It only clears if the user manually
  clears their browser data, or if you call localStorage.removeItem() in code.

  What your app stores
  Two keys in localStorage:
  "ratings"  →  { "1": 4, "2": 5 }
  "reviews"  →  { "1": "Great film.", "3": "A classic." }
  Each is a JSON string. The keys inside are movie IDs, so each film has its own
   slot.

  What happens when you rate a film
  1. You click a star
  2. handleRate runs — it reads the current ratings object out of localStorage,
  sets stored[id] = star, then writes the whole thing back
  3. React state updates so the stars reflect the new rating immediately

  What happens on reload
  The useState for both rating and review uses an initializer function:
  const [rating, setRating] = useState(() => {
    const stored = JSON.parse(localStorage.getItem('ratings') || '{}');
    return stored[id] || 0;
  });
  That function runs once when the component first mounts. It reads from
  localStorage before React renders anything, so the stars and review text are
  already correct the moment the page loads — there's no flicker or loading
  state.

  The limitation
  Since it's stored in the browser, it's device-specific. If you open the site
  on a different computer or a different browser, the ratings and reviews won't
  be there. That's fine for a personal tracker — it's by design. A backend
  database would be needed to sync across devices, but that's well beyond the
  scope of this project.
